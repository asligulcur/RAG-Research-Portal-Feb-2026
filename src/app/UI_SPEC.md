# Personal Research Portal — UI & UX Specification (v2)

> **Instructions to Agent**: This is the complete UI and UX specification for the AG Research Portal. Follow this document exactly for all layout, page structure, interaction flows, and component rendering. Do NOT improvise layouts or add features not described here. The design system is already defined in `styles.py` — this spec tells you HOW to use it. When implementing, match the wireframes and flows precisely.

---

## 0. Design Philosophy

This portal should feel like an internal research tool at McKinsey or Goldman Sachs — **authoritative, precise, zero fluff**. Every element earns its space. The brand is:

- **Light background, dark navy authority** — white/off-white canvas with navy (#0F2B46) as the power color
- **Gold (#C5A028) as the signature accent** — used sparingly for emphasis, active states, and brand marks. Gold is the "this matters" signal. Overuse kills it.
- **Royal blue (#1A4B7A) for interactive elements** — links, focus states, citation chips
- **Typography is sharp** — tight letter-spacing on headings, generous line-height on body. No small text. Nothing under 13px except legal-style footnotes.
- **White space is structural** — panels breathe. Cards don't crowd. Sections are visually separated by space, not by heavy dividers.
- **Data density when appropriate** — evidence tables, evaluation dashboards, and source libraries can be dense. But the Search & Ask page should feel spacious and focused.

**Tone**: If someone screenshots this app, it should look like it was built by a design-aware engineering team, not generated by an AI.

### Reference Designs (study these for visual direction)

Before implementing any page, look at these products for the target feel. You are not copying their features — you are matching their **visual authority, information density, and professional restraint**.

- **Elicit.com** (elicit.com) — The research workflow: query → evidence cards → source metadata. Study how they present retrieved papers with structured fields. Our Evidence Explorer and Source Library should feel like this.
- **Perplexity.ai** — The answer panel: clean answer with inline citations that link to sources below. Our Search & Ask answer panel should have this citation-to-evidence flow.
- **Bloomberg Terminal** (dark sidebar, dense data panels) — The information hierarchy: a dark navigation rail on the left, high-density data views in the main panel, status indicators using color coding. Our Evaluation Dashboard should feel like a lighter, cleaner version of this.
- **McKinsey reports** (mckinsey.com/insights) — The typography and whitespace: large confident headings, generous margins, gold/navy accent palette, charts that breathe. Our Artifact Generator outputs (especially Synthesis Memo) should read like a McKinsey brief.

**What to take from each**: layout structure and information density, NOT specific UI components. We're building in Streamlit, so translate the *feel* into Streamlit patterns.

## 0.5. User Journeys & End-to-End Flows

These are the primary user journeys through the portal. Every page transition, button, and UI element exists to serve one of these flows. **If a feature doesn't appear in a journey below, it's not needed for MVP.**

The agent must ensure these flows are seamless — meaning every action in a step naturally leads to the next step with no dead ends, no confusion about what to do next, and no need to manually navigate unless the user wants to.

---

### Journey 1: "Answer My Research Question" (Primary — 80% of usage)

This is the core loop. A researcher has a question and wants a grounded, cited answer.

```
STEP 1: Land on Search & Ask (default page)
  → User sees: empty query input with placeholder example question
  → User sees: subtle prompt below input: "Try: What are the advantages of small language models?"
  → The page is clean and focused — nothing else competes for attention

STEP 2: Type and submit a question
  → User types query into the input field
  → User clicks "Ask" button (or presses Enter)
  → IMMEDIATE feedback: spinner appears with "Retrieving evidence from 15 sources..."
  → User knows the system is working. No ambiguity.

STEP 3: Read the answer
  → Spinner disappears. Answer panel appears with gold left-border.
  → Answer is 1-3 paragraphs with inline citation chips [Smith2023] styled in royal blue.
  → Below the answer: a status strip shows grounding quality
     - Green "● GROUNDED" if well-supported
     - Amber "● PARTIALLY GROUNDED" if some claims lack evidence
     - Red "● NOT GROUNDED" if evidence is weak
  → User can READ the answer and immediately assess its trustworthiness.

STEP 4: Inspect the evidence
  → User scrolls down (or the page auto-scrolls to) "Retrieved Evidence" section
  → Evidence cards appear, sorted by relevance score
  → Each card shows: source_id + chunk_id, relevance score, text snippet, metadata
  → User can verify: "Does this snippet actually support that claim?"
  → Clicking a citation chip in the answer scrolls to / highlights the matching evidence card

STEP 5: Take action on the results
  → Below the answer panel, user sees three action buttons:
     [Save Thread]  →  saves query + answer + evidence to Research Threads page
     [Generate Artifact]  →  navigates to Artifact Generator with this query + evidence pre-loaded
     [Copy Answer]  →  copies the answer text (with citations) to clipboard
  → These are the ONLY next actions. No decision fatigue.

STEP 6 (optional): Refine and re-ask
  → The query input remains at the top of the page, still containing the original query
  → User can edit the query and re-submit
  → New results replace old results (no chat history stacking — this is search, not chat)
```

**Agent implementation notes for Journey 1:**
- The query input must persist after submission (don't clear it)
- Results must fully replace previous results (no appending)
- The "Generate Artifact" button must pass the current query AND retrieved evidence IDs to the Artifact Generator page via `st.session_state`
- Citation chip clicks should use anchor links or `st.session_state` to scroll/highlight the matching evidence card
- The page should never show both a spinner and old results simultaneously — clear old results when a new query starts

### Journey 2: "Generate a Research Deliverable" (Secondary — artifact creation)

The researcher has gathered evidence and now wants to produce a structured artifact for a report, presentation, or further analysis.

```
STEP 1: Arrive at Artifact Generator
  → User navigates here via sidebar OR via "Generate Artifact" button from Search & Ask
  → If arriving from Search & Ask: query and source scope are pre-filled
  → If arriving from sidebar: fields are empty, ready for input

STEP 2: Configure the artifact
  → User selects artifact type: Evidence Table / Annotated Bibliography / Synthesis Memo
  → User reviews or edits the research question
  → User optionally narrows source scope (select specific sources or "All")
  → User sets max entries (default 10)
  → Configuration panel is a single, contained widget — no scattered controls

STEP 3: Generate
  → User clicks "Generate Artifact" (primary button)
  → Spinner: "Generating evidence table... analyzing 15 sources"
  → Generated artifact renders below the config panel inside a gold-bordered panel

STEP 4: Review the artifact
  → User reads the generated content
  → For Evidence Table: a clean data table with Claim | Evidence | Citation | Confidence | Notes
  → For Annotated Bibliography: source cards with Key Claim, Method, Limitations, Relevance
  → For Synthesis Memo: formatted prose with inline citations and a References section
  → Every citation in the artifact is verifiable against the corpus

STEP 5: Export
  → Below the artifact, export buttons appear: [Download CSV] [Download Markdown] [Download PDF]
  → User clicks the desired format → file downloads immediately
  → The exported file is also saved to outputs/artifacts/ for later access via Export Center
  → User is done. They have their deliverable.
```

**Agent implementation notes for Journey 2:**
- Pre-filling from Search & Ask is critical. If `st.session_state` has `last_query` and `last_evidence_ids`, use them.
- The artifact type radio button should update a preview description below it explaining what that artifact contains
- Export buttons only appear AFTER an artifact is generated (not before)
- Generated artifacts auto-save to disk — user shouldn't need to explicitly save

### Journey 3: "Evaluate My System's Quality" (Grading-critical)

The researcher wants to see how well the RAG system performs across a set of test queries.

```
STEP 1: Navigate to Evaluation Dashboard
  → If no evaluation has been run: show empty state with a single prominent button: [Run Evaluation]
  → If evaluation has been run before: show the most recent results immediately

STEP 2: Run evaluation
  → User clicks "Run Evaluation"
  → Progress bar appears: "Running evaluation set... 5/20 queries complete"
  → Progress updates in real-time as each query is processed
  → When complete: results render below

STEP 3: Review aggregate metrics
  → Top of page: 4 metric cards in a row
     Groundedness | Citation Precision | Relevance | Query Count
  → Each metric value is color-coded (green > 0.7, amber 0.5-0.7, red < 0.5)
  → User gets an instant read on system quality

STEP 4: Drill into per-query results
  → Below metrics: a sortable table showing each query with its individual scores
  → User can sort by any metric column to find worst-performing queries
  → Clicking a row could expand to show the full query + answer + evidence (stretch, not required)

STEP 5: Review failure cases
  → Below the table: 3+ failure case cards
  → Each card shows: query, failure tag (HALLUCINATION / MISSING EVIDENCE / WRONG CITATION / OVERCONFIDENT), what went wrong, the retrieved evidence vs. generated answer, and a suggested fix
  → These are the most important part for the grading report

STEP 6: Export
  → [Export Report] button generates a markdown or CSV summary of all metrics + failure cases
  → Saved to logs/eval_runs/ and available in Export Center
```

**Agent implementation notes for Journey 3:**
- The progress bar is important — evaluation runs 20+ queries and takes time
- Failure cases should be auto-identified by the evaluation script (low scores, citation mismatches)
- The evaluation should be re-runnable — running it again replaces previous results
- Eval logs must be machine-readable JSON (for the grading requirement)

### Journey 5: "Review Past Research" (Thread management)

```
STEP 1: Navigate to Research Threads

STEP 2: Expand a thread
  → Click "View Thread" → expander opens showing:
```

### Cross-Cutting Flow Rules (apply to ALL journeys)

1. **No dead ends**: Every page must have a clear next action. If there's nothing to show, the empty state tells the user exactly what to do and where to go.

2. **Breadcrumb context**: When arriving at a page from another page (e.g., Search & Ask → Artifact Generator), show a subtle context line: "Based on your query: [truncated query text]" so the user knows what data they're working with.

3. **Navigation never loses state**: Switching pages via the sidebar does NOT clear the current results on Search & Ask. When the user returns, their last query and results are still there.

4. **Consistent action patterns**:
   - Primary button (navy) = "do the main thing" (Ask, Generate, Run Evaluation)
   - Secondary buttons (white/bordered) = "do a follow-up thing" (Save, Export, Copy, View)
   - Destructive buttons (red text, no fill) = Delete only

5. **Progressive disclosure**: Filters and advanced options are always collapsed by default (`st.expander`). The default path is: type question → get answer → take action. Power users can expand filters when needed.

6. **One primary action per page**: Each page has exactly one "main thing to do":
   - Search & Ask → Ask a question
   - Artifact Generator → Generate an artifact
   - Evaluation Dashboard → Run evaluation
   - Source Library → Browse sources (passive — no primary action)
   - Research Threads → Review threads (passive)
   - Export Center → Download files (passive)

|-------|---------|---------------|

| `--green` (#0D9668) | High confidence, success states, "grounded" indicators | — |
| `--amber` (#D49717) | Medium confidence, warnings, "partially grounded" | — |
| `--error-red` (#CC3333) | Low confidence, errors, "not grounded", missing evidence | — |

### The Gold Rule
Gold appears in exactly these places and nowhere else:
1. Left border on the executive summary / answer panel (5px solid)
2. Underline on widget section titles (2px solid)
3. Active navigation indicator in sidebar (left border)
4. The "◆" brand mark next to the app title

If you're about to use gold somewhere not on this list, don't.

### Typography Scale (already in CSS, reference for components)
- Page title (h1): 44px / 800 weight / navy / -0.5px tracking
- Section heading (h2): 32px / 700 / navy

- Metadata: 17px / 500 / slate
- Pill/badge text: 12-13px / 700 / uppercase / 0.5px tracking

**Active state for sidebar buttons**: When a nav item is active, apply:
```css
border-left: 3px solid #C5A028 !important;
background: rgba(255,255,255,0.07) !important;
color: #FFFFFF !important;
```

**Implementation**:
```python
def nav_button(label, page_key):
    """Render a sidebar nav button. Returns True if clicked."""
    is_active = st.session_state.get("current_page") == page_key
    if st.sidebar.button(label, key=f"nav_{page_key}", use_container_width=True):
        st.session_state.current_page = page_key
        st.rerun()
    return is_active
```

Then conditionally inject CSS to highlight the active button. Alternatively, use `st.markdown` with custom HTML for full control.

**No emoji in nav labels.** This is a professional tool. Use plain text for nav items.

#### Citation Rendering:
```python
import re

def render_citations(text: str) -> str:
    """Replace (SourceID) or (SourceID, chunk_XX) patterns with styled chips."""
    pattern = r'\((\w+(?:\d{4})(?:,\s*chunk_\d+)?)\)'
    def _chip(m):
        return f'<span class="citation">{m.group(1)}</span>'
    return re.sub(pattern, _chip, text)

# In Streamlit:
st.markdown(
    f'<div class="executive-summary">{render_citations(answer)}</div>',
    unsafe_allow_html=True
)
```

## 4. Trust & Edge Case Behaviors (Required — not optional)

These behaviors are grading criteria. Implement all of them.

1. **No evidence found** (0 relevant chunks or all below threshold):
   - Render `.exec-status-strip` with `.exec-status-tone-warning`
   - Message: "No relevant evidence found in the corpus for this query. Consider rephrasing or broadening your search terms."
   - Do NOT generate an answer. Show only the status strip.

2. **Partial evidence** (answer contains ungrounded claims):
   - Append to the answer panel: `⚠ Some claims in this answer may not be fully supported by the corpus. Claims without citations should be verified independently.`
   - Use amber color for the warning.

3. **Conflicting evidence**:
   - Inline in the answer: `⚡ Sources disagree: [SourceA] argues X while [SourceB] argues Y.`

4. **Fabricated citations**:
   - Before rendering, validate every citation ID against the data manifest.
   - If a citation ID doesn't exist, strip it and append: `⚠ One or more citations were removed because they could not be resolved to a source in the corpus.`

5. **Empty corpus / not indexed**:
   - Show info-toned status strip: "No sources have been indexed yet. Add sources to data/raw/ and run the ingestion pipeline."

### Streamlit Component Selection Rules

Use this exact mapping. Do not substitute alternatives.

| UI Need | Use This | NOT This |
|---------|----------|----------|
| Page navigation | `st.sidebar.button` per item + `st.session_state` | `st.radio`, `st.selectbox`, `st.page_link`, multipage |
| Query input | `st.text_input` + `st.button` side by side via `st.columns([5,1])` | `st.chat_input`, `st.text_area` |
| Filters (collapsed) | `st.expander("Filters")` containing `st.columns` with `st.selectbox` / `st.slider` | Filters always visible |
| Answer display | `st.markdown(html, unsafe_allow_html=True)` | `st.write`, `st.text`, `st.chat_message` |
| Evidence cards | `st.markdown(html, unsafe_allow_html=True)` per card | `st.json`, `st.code`, `st.info` |
| Data tables | `st.dataframe` with `column_config` | `st.table` (no sorting), raw HTML tables |
| Metric display | Custom HTML `.metric-card` via `st.markdown` | `st.metric` (can't style to match) |
| Downloads | `st.download_button` | Writing files and showing paths |
| Loading feedback | `st.spinner` for short ops, `st.progress` for batch ops | No feedback, `st.toast` |
| Expandable content | `st.expander` | Custom JS toggles |
| Layout columns | `st.columns` | CSS grid (unreliable in Streamlit) |
| Page section spacing | `st.markdown("<br>", unsafe_allow_html=True)` or `st.divider()` | Multiple `st.write("")` |

### Component Build & Test Order

Build components in this order. After each, test with sample data on a scratch page before integrating.

1. `render_citations()` — Pure function, no Streamlit dependency. Unit test it.
2. `render_answer_panel()` — Depends on `render_citations()`. Test with a hardcoded answer string.
3. `render_evidence_card()` — Test with a sample chunk dict.
4. `render_status_strip()` — Test all three tones (success, warning, info).
5. `render_metric_card()` — Test with sample values in all three color ranges.
6. `render_source_card()` — Test with a sample manifest row.
7. `render_thread_card()` — Test with a sample thread dict.
8. `render_empty_state()` — Test on a blank page.
9. Sidebar navigation — Test page switching with 3 dummy pages.

Only after ALL components render correctly should you assemble them into full pages.

## 10. What NOT to Do

- ❌ No emoji in navigation labels or section headers.
- ❌ No `st.metric` — use custom `.metric-card` HTML for visual consistency.
- ❌ No chatbot/message-style interface. This is a search tool, not a conversation.
- ❌ No separate CSS files. All styles live in `styles.py` as `AG_RESEARCH_CSS`, injected once at the top of `app.py`.
- ❌ No Streamlit multipage app. Single `app.py` with sidebar routing.
- ❌ No features beyond this spec. Get the MVP right first.
- ❌ No gold on backgrounds, buttons, or large areas. Gold is accent only (borders, underlines, brand mark).
- ❌ No bright/saturated colors anywhere except the defined green/amber/red status indicators.
- ❌ Do not override `styles.py` with inline styles that contradict the design system. If you need a new style, add it to `styles.py` following the existing patterns.
- ❌ No placeholder / lorem ipsum text. Use realistic research examples from the user's domain.
